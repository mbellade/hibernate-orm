= Envers Rewrite Plan
:toc:
:toclevels: 3

== Overview

The `hibernate-envers` module currently relies on hbm.xml-contributed runtime mapping objects to add audit entity
mappings to the SessionFactory, and provides a dedicated read API (`AuditReader`, `AuditQuery`) on top of those
mappings.

Hibernate Core 7.4 introduced `@org.hibernate.annotations.Audited`, which integrates audit trail persistence directly
into mutation coordinators and provides transparent read access via `atTransaction(txId)` sessions.
The write side is functional (though not yet feature-complete). This document plans the path forward for both
**write-side** completeness and the **read side** — migrating envers capabilities to hibernate-core.

We do **not** need to re-implement identical APIs. The goal is to provide equivalent functionality with a clear
migration path, leveraging the simpler `atTransaction` session paradigm wherever possible.

The ultimate objective for this migration is avoiding the need to use the old `hbm.xml` mapping format,
as that is deprecated and will be removed in the upcoming Hibernate major version.

== Current State

=== hibernate-core `@Audited` (since 7.4, @Incubating)

Write::
Fully integrated into mutation coordinators.
INSERT/UPDATE/DELETE automatically create audit rows in `*_aud` tables with:
- Transaction ID column (`REV`, configurable) — application-supplied via `TransactionIdentifierService`
- Modification type column (`REVTYPE`, configurable) — 0=ADD, 1=MOD, 2=DEL

Read::
Open a session with `sessionFactory.withOptions().atTransaction(txId).open()`, then use **standard Hibernate APIs**:
`session.find()`, HQL, criteria. Audit table filtering is applied transparently via `AuditMappingImpl.applyPredicate()`
which builds a subquery to find `MAX(REV)` from the audit table where `REV \<= suppliedTxId` and `REVTYPE != 2` (DEL).

Key differences from envers::
- No separate "audit entity" mappings — same entity class, transparently reads from audit table
- No revision entity (REVINFO) — transaction IDs are opaque, application-supplied (any comparable type)
- No custom query DSL — everything goes through the normal session
- Attribute exclusion via `@Audited.Excluded` (equivalent to `@NotAudited`)

Key files::
- `org.hibernate.annotations.Audited` — annotation
- `org.hibernate.metamodel.mapping.AuditMapping` — metamodel interface
- `org.hibernate.metamodel.mapping.internal.AuditMappingImpl` — predicate application for reads
- `org.hibernate.persister.state.internal.AuditStateManagement` — write-side coordinators
- `org.hibernate.boot.model.internal.AuditHelper` — boot-time table creation
- `org.hibernate.engine.creation.CommonBuilder#atTransaction(Object)` — session builder method
- `org.hibernate.temporal.AuditEntityTest` — test demonstrating current capabilities

=== hibernate-envers (existing)

Write::
Event listeners (`EnversPostInsertEventListenerImpl`, etc.) capture DML operations, create work units
(`AddWorkUnit`, `ModWorkUnit`, `DelWorkUnit`), and queue them via `AuditProcess` for insertion at transaction
commit. Audit entity mappings are contributed via `AdditionalMappingContributorImpl` using hbm.xml JAXB generation.

Read::
Dedicated API surface:
- `AuditReaderFactory.get(session/em)` → `AuditReader`
- `AuditReader.find(Class, id, revision)` — point-in-time entity retrieval
- `AuditReader.getRevisions(Class, id)` — revision history
- `AuditReader.getRevisionDate(revision)` / `getRevisionNumberForDate(date)` — revision ↔ time mapping
- `AuditReader.createQuery()` → `AuditQueryCreator` → `AuditQuery` — rich query DSL
- `AuditEntity` — static factory for criteria/projections (`property()`, `revisionNumber()`, `revisionType()`, etc.)
- `CrossTypeRevisionChangesReader` — query all entity types changed in a revision

Key envers read patterns from tests::
- `forEntitiesAtRevision(Class, rev)` — bulk entity state at a revision (with filters, ordering, pagination)
- `forRevisionsOfEntity(Class, selectEntitiesOnly, selectDeleted)` — full revision history with snapshots
- `forEntitiesModifiedAtRevision(Class, rev)` — only entities changed IN that revision
- `AuditEntity.property("field").hasChanged()` / `hasNotChanged()` — modified flag queries
- `AuditEntity.relatedId("relation").eq(id)` — relationship-based filters
- Aggregate projections (`max()`, `min()`, `count()`, `function()`)
- Association traversal (`traverseRelation()`)

== Philosophical Difference

[cols="1,2,2"]
|===
| | Envers | Core `@Audited`

| Read paradigm | Dedicated audit-specific API | Standard session APIs with temporal context
| Query model | Custom DSL (AuditQuery/AuditEntity) | HQL/Criteria/find() — same as regular entities
| Entity identity | Separate "audit entity" mappings (hbm.xml) | Same entity class, reads from audit table transparently
| Revision model | First-class RevisionEntity with auto-increment | Opaque transaction ID (app-supplied)
|===

The core approach is fundamentally simpler: **"open a session at a point in time, use normal Hibernate."**
This eliminates the need for most of the envers query DSL, since HQL already provides filtering, ordering,
projections, joins, and aggregations.

[[not-audited-target-behavior]]
=== Associations to Non-Audited Targets: Behavioral Differences

Core auditing handles mixed audited/non-audited entity relationships implicitly — no annotation
needed (unlike envers' `@Audited(targetAuditMode = NOT_AUDITED)`). The FK is stored in the audit
table and used to load the target from the current table. However, there are behavioral differences
with envers when the non-audited target entity has been **deleted**:

[cols="1,2,2"]
|===
| Scenario | Envers behavior | Core behavior

| **Eager / `optional=true`**
| `targetNotFoundAction` controls: ERROR (default) throws, IGNORE returns null
| LEFT JOIN resolves to null silently (standard Hibernate semantics)

| **Lazy + deleted target**
| Proxy created; on access, `targetNotFoundAction` controls: ERROR throws `EntityNotFoundException`,
  IGNORE eagerly loads and returns null (no proxy)
| Proxy created; on access, throws `EntityNotFoundException` (standard Hibernate proxy behavior)

| **`optional=false` + deleted target**
| ERROR throws, IGNORE returns null
| INNER JOIN fails to load the owning entity itself (returns null from `find()`)
|===

**Root cause**: envers had its own entity reconstruction pipeline (field-by-field mapping from audit
row data, separate per-association loading via `ToOneEntityLoader`). Core reuses standard Hibernate
entity loading with JOIN fetching, so association behavior follows standard Hibernate semantics.

**Note**: audit tables have no FK constraints (same as envers), so dangling FKs are always possible.
For audited-to-audited associations, envers' `AuditReaderImpl.find()` silently returned null for
missing targets (`NoResultException` caught). Core's JOIN-based approach handles this via the
temporal predicate + LEFT JOIN.

**Migration options**:

1. **Recommended**: migrating users should annotate audited-to-non-audited lazy associations with
   `@NotFound(action = IGNORE)` or `@NotFound(action = EXCEPTION)` to explicitly declare the
   expected behavior when the target is deleted.
2. **Future**: core could implicitly default to `@NotFound(EXCEPTION)` or `@NotFound(IGNORE)` for
   associations from audit tables to non-audited entities, matching envers' `targetNotFoundAction`
   default. This would require changes to how audit table references handle association fetching.

== Write-Side Gap Analysis

The core `@Audited` write infrastructure handles the basic audit trail (INSERT/UPDATE/DELETE → audit row with
TX ID + modification type), but envers has many additional write-side features that need consideration.

=== Custom Revision Entity

Envers provides a first-class revision entity model:

- `@RevisionEntity` marks a custom entity as the revision tracker
- `@RevisionNumber` / `@RevisionTimestamp` mark its fields
- `RevisionListener.newRevision(revisionEntity)` callback lets users populate custom fields (user, comment, etc.)
- `EntityTrackingRevisionListener.entityChanged(...)` notifies per-entity change
- The revision entity is persisted automatically; its generated ID becomes the `REV` value in all audit rows
- Built-in defaults: `DefaultRevisionEntity`, `SequenceIdRevisionEntity`, tracking variants with `@ModifiedEntityNames`

Core currently has `TransactionIdentifierService` with a plain `Supplier<?>` — no session access, no persistence.

**Proposed approach**: Enhance `TransactionIdentifierService` to support session-aware revision entity creation.
See <<Custom Revision Entity Support>> section below.

== Deferred Audit Writes (Transaction-Scoped Work Unit Queue)

=== Problem

Core's `@Audited` writes audit rows **inline** during flush, directly from mutation coordinators
(`InsertCoordinatorAudit`, `UpdateCoordinatorAudit`, `DeleteCoordinatorAudit`). This breaks when
the same entity is modified across multiple flushes within a single transaction:

[source,java]
----
tx.begin();
entity.setName("a");
session.flush();           // → audit INSERT (REV=5, REVTYPE=MOD)
entity.setName("b");
session.flush();           // → audit INSERT (REV=5, REVTYPE=MOD) → PK violation!
tx.commit();
----

The audit table PK is `(REV, id)`, and both flushes produce the same `REV` (same transaction).

Envers solved this by **never writing audit rows at flush time**. Instead:

1. Event listeners create **work units** (ADD/MOD/DEL) and queue them in `AuditProcess`
2. If the same entity `(entityName, id)` is touched again, the new work unit **merges** with
   the existing one via a visitor pattern dispatch
3. All audit rows are written at `beforeTransactionCompletion` — one row per entity per transaction

=== Envers Merge Matrix

[cols="1,1,1,3"]
|===
| Existing | New | Result | Notes

| ADD | MOD | ADD | Latest state, merged modified flags
| ADD | DEL | _(cancel)_ | Entity created and deleted in same tx — no audit row
| MOD | MOD | MOD | Latest state from second MOD, old state from first MOD
| MOD | DEL | DEL | Deletion wins
| DEL | ADD | MOD | Re-use of deleted entity — compare states
|===

=== Proposed Design for Core

==== Audit work queue

Introduce a transaction-scoped `AuditWorkQueue` (or similar), accessible from the session.
Mutation coordinators enqueue audit entries instead of writing them directly.

[source]
----
┌─────────────────────────┐
│  Mutation Coordinator   │
│  (Insert/Update/Delete) │
└──────────┬──────────────┘
           │ enqueue(entityName, id, state, modificationType)
           ▼
┌─────────────────────────┐
│   AuditWorkQueue        │
│   (transaction-scoped)  │
│                         │
│  Map<(entity, id),      │
│       QueuedAuditEntry> │
│                         │
│  merge logic:           │
│   ADD+MOD → ADD         │
│   MOD+MOD → MOD         │
│   ADD+DEL → remove      │
│   MOD+DEL → DEL         │
└──────────┬──────────────┘
           │ beforeTransactionCompletion
           ▼
┌─────────────────────────┐
│  Audit Row Writer       │
│  - INSERT audit rows    │
│  - UPDATE REVEND (val.) │
└─────────────────────────┘
----

==== Key design points

1. **Queue keyed by `(entityName, id)`** — at most one pending entry per entity per transaction.
   When a second modification arrives for the same entity, merge with the existing entry.

2. **Merge semantics** — follow envers' matrix. The merge only needs to track the latest entity
   state and the effective modification type. Modified flags (`_MOD` columns) are deferred, so
   the merge is simpler than envers' (just state + type).

3. **Transaction completion hook** — register a `BeforeTransactionCompletionProcess` (or use
   `TransactionCompletionCallbacks`) to flush the queue. All audit INSERTs and REVEND UPDATEs
   happen here, in one batch.

4. **Coordinator changes** — `insertAuditRow()` becomes `enqueueAuditEntry()`. The cached
   `MutationOperationGroup` for INSERT (and REVEND UPDATE) stays on `AbstractAuditCoordinator`
   but is used at transaction completion, not at flush time.

5. **Collection audit rows** — collection coordinators (`InsertRowsCoordinatorAudit`, etc.)
   also need to enqueue rather than write inline. Collection entries are keyed differently
   (by collection key + element/index, not entity id).

6. **ValidityAuditStrategy integration** — the REVEND UPDATE runs at transaction completion,
   after all audit INSERTs. Since there's exactly one audit row per entity per transaction,
   the UPDATE targets exactly one previous row. No `REV <> ?` clause needed if we ensure
   ordering (UPDATE previous row, then INSERT new row).

==== What stays the same

- Boot-time schema creation (`AuditHelper`) — unchanged
- Runtime mapping (`AuditMappingImpl`, `TableAuditInfo`) — unchanged
- Read-side predicates (MAX subquery, validity range) — unchanged
- `MutationOperationGroup` caching — the groups are still built once and reused

* [x] Define `AuditWorkQueue` SPI — transaction-scoped, keyed by `EntityKey` (entities)
  and `CollectionKey` (collections)
* [x] Define merge logic for `ModificationType` transitions

* [x] Refactor `AbstractAuditCoordinator` → `enqueueAuditEntry()` + `writeAuditRow()` (via `AuditWriter` SPI)

== Custom Revision Entity Support

=== Problem

Envers users define custom revision entities to store per-transaction metadata:

[source,java]
----
@RevisionEntity(MyRevisionListener.class)
@Entity
public class MyRevision {
    @RevisionNumber @Id @GeneratedValue
    private int id;

    @RevisionTimestamp
    private Instant timestamp;

    private String username;
    private String comment;
}

public class MyRevisionListener implements RevisionListener {
    @Override
    public void newRevision(Object revisionEntity) {
        MyRevision rev = (MyRevision) revisionEntity;
        rev.setUsername(SecurityContext.getCurrentUser());
        rev.setComment(AuditContext.getComment());
    }
}
----

The revision entity is:

1. Instantiated and populated (via `RevisionListener`) once per transaction
2. Persisted to the database (generating an ID)
3. Its generated ID used as the `REV` value in all audit rows for that transaction

=== Current Core Limitation

`TransactionIdentifierService` provides a `Supplier<?>` that returns an opaque ID.
It has **no access to a Session** — it cannot persist anything:

[source,java]
----
public interface TransactionIdentifierService extends Service {
    Class<?> getIdentifierType();
    Supplier<?> getIdentifierSupplier();
    // ...
}
----

=== Proposed Solution

Enhance `TransactionIdentifierService` to support a session-aware supplier that can persist a revision entity
and return its generated identifier. Two approaches:

==== Option A: Richer supplier contract (recommended)

Replace the plain `Supplier<?>` SPI with a dedicated `TransactionIdentifierSupplier<T>` that always
receives the session. Implementors who don't need the session simply ignore it.

[source,java]
----
/**
 * A supplier of transaction identifiers.
 * Implementations may persist a revision entity and return its generated ID.
 */
@Incubating
public interface TransactionIdentifierSupplier<T> {
    /**
     * Called once per transaction to obtain the transaction identifier.
     * The implementation may persist a revision entity using the provided session.
     *
     * @param session the current session (can be used to persist entities)
     * @return the transaction identifier (e.g. the generated ID of a revision entity)
     */
    T getTransactionIdentifier(SharedSessionContract session);
}
----

**Status**: [green]#DONE# — `TransactionIdentifierService.getIdentifierSupplier()` now returns
`TransactionIdentifierSupplier<?>`. Plain `Supplier<?>` instances are auto-wrapped for backward compatibility.

Usage example:

[source,java]
----
@Entity
public class RevisionInfo {
    @Id @GeneratedValue
    private Long id;
    private Instant timestamp;
    private String username;
    // getters/setters
}

public class RevisionInfoSupplier implements TransactionIdentifierSupplier<Long> {
    @Override
    public Long getTransactionIdentifier(SharedSessionContract session) {
        var rev = new RevisionInfo();
        rev.setTimestamp(Instant.now());
        rev.setUsername(SecurityContext.getCurrentUser());
        session.persist(rev);
        session.flush(); // ensure ID is generated
        return rev.getId();
    }
}
----

Configuration:
[source,properties]
----
hibernate.temporal.transaction_id_supplier=com.example.RevisionInfoSupplier
----

==== Option B: Callback-based (RevisionListener-like)

Keep `Supplier<?>` but add a callback interface similar to envers' `RevisionListener`:

[source,java]
----
@Incubating
public interface AuditTransactionListener {
    /**
     * Called once per transaction before audit rows are written.
     * Can persist revision metadata using the provided session.
     */
    void onAuditTransaction(SharedSessionContractImplementor session, Object transactionId);
}
----

This separates ID generation from revision entity persistence but adds complexity.

==== Recommendation

**Option A** is cleaner — a single extension point that handles both ID generation and revision entity
persistence in one call. It maps directly to the envers `RevisionInfoGenerator` pattern and provides
a straightforward migration path:

[cols="2,3"]
|===
| Envers | Core

| `@RevisionEntity` + `@RevisionNumber` + `@RevisionTimestamp`
| Regular `@Entity` — user defines structure freely

| `RevisionListener.newRevision(revisionEntity)`
| Logic inside `TransactionIdentifierSupplier.getTransactionIdentifier(session)`

| `EntityTrackingRevisionListener.entityChanged(...)`
| Can be covered by a separate callback or event listener (Phase 2)

| Automatic revision entity persistence
| `RevisionEntitySupplier` handles persistence via child `StatelessSession`

| Revision number = generated ID
| Return value of `getTransactionIdentifier()` = transaction ID used in audit rows
|===

==== 1.3 RevisionType exposure and audit history reads

`REVTYPE` and `REV` should be accessible during reads, but not as synthetic entity properties.
Instead, custom HQL functions provide clean access:

* `transactionId(e)` — resolves to the `REV` column of the audit table
* `modificationType(e)` — resolves to the `REVTYPE` column of the audit table

These functions only work when querying against the audit table (temporal/audit mode).

===== All-revisions mode via `AuditLog.ALL_REVISIONS`

In addition to `atTransaction(txId)` (point-in-time snapshot with MAX subquery + REVTYPE != DEL filter),
passing `AuditLog.ALL_REVISIONS` to `atTransaction()` provides raw access to the audit table:

* Reads from the audit table (same as `atTransaction`)
* Does NOT apply the MAX(REV) subquery — returns ALL audit rows
* Does NOT filter out REVTYPE = DEL — deleted entity rows are included
* Enables the `transactionId()` and `modificationType()` HQL functions

This mode solves two problems at once:

1. **History queries** (equivalent to envers' `forRevisionsOfEntity`):
+
[source,java]
----
try (var s = sf.withOptions()
        .atTransaction(AuditLog.ALL_REVISIONS).open()) {
    var history = s.createSelectionQuery("""
            select e, r, modificationType(e)
            from MyEntity e join RevisionEntity r
                on transactionId(e) = r.id
            where e.id = :id
            order by transactionId(e)
            """, Object[].class)
        .setParameter("id", entityId)
        .getResultList();
    // Each result: [entity, revisionEntity, ModificationType]
}
----

2. **Deleted entity access** (currently `find()` returns null for deleted entities):
+
[source,java]
----
try (var s = sf.withOptions()
        .atTransaction(AuditLog.ALL_REVISIONS).open()) {
    var deleted = s.createSelectionQuery("""
            select e from MyEntity e
            where e.id = :id and modificationType(e) = 2
            """, MyEntity.class)
        .setParameter("id", entityId)
        .getSingleResultOrNull();
    // Returns the entity state at the moment of deletion
}
----

NOTE: In all-revisions mode, `TemporalEntityKey` (a sealed subtype of `EntityKey` that includes the
transaction identifier in `equals`/`hashCode`) ensures the `PersistenceContext` naturally isolates
the same entity at different points in time. Each audit row produces a distinct entity instance
keyed by `(entityClass, id, txId)`. The base `EntityKey` remains lightweight (no extra fields) —
`TemporalEntityKey` is only allocated for entities loaded from audit tables.
All temporal entities are forced `Status.READ_ONLY` (via `EntityKey.isTemporal()`) — no dirty
tracking, no flush, no 2LC caching.

===== Audited associations

Associations between `@Audited` entities work transparently in both point-in-time and all-revisions modes:

* **Point-in-time**: The session's `temporalIdentifier` is fixed — association loads automatically
  use the audit table at the correct revision.
* **All-revisions (select-fetch)**: `AbstractEntityResultGraphNode` implicitly selects the `REV_TXN`
  column for audited entities (like discriminator columns). `EntityInitializerImpl` reads this per
  row and sets it as the temporal identifier on `LoadQueryInfluencers`, so association loaders
  generate SQL with the correct temporal predicate.
* **All-revisions (join-fetch)**: `AuditMappingImpl.applyPredicate()` detects all-revisions mode
  and builds a correlated predicate: `REV = (SELECT MAX(REV) ... WHERE REV \<= parent.REV)`.
  The parent's REV column is found by walking `navigablePath.getParent()` and checking for
  `EntityValuedModelPart` with an audit mapping. Works for nested associations at any depth.
* **Lazy to-one proxies**: `AbstractLazyInitializer` captures the `temporalIdentifier` from the
  session at proxy creation time. When the proxy is later initialized via `immediateLoad()`, the
  captured revision is temporarily set on the session's `LoadQueryInfluencers` (then restored).
  This mirrors envers' `ToOneDelegateSessionImplementor` pattern but without session wrapping.
* **`TemporalEntityKey`**: Sealed subtype of `EntityKey` that includes the transaction identifier
  in `equals()`/`hashCode()`, preventing PC deduplication across revisions (e.g. Author#20@tx201 ≠
  Author#20@tx202). The base `EntityKey` has no extra fields — zero memory overhead for non-auditing apps.
* **Null associations in ALL_REVISIONS**: Users must use `left join fetch` (not inner `join fetch`)
  when fetching nullable associations in ALL_REVISIONS mode, otherwise revisions where the
  association is null will be silently excluded. This is standard HQL join semantics.

===== Per-row temporal identifier: initializer ordering considerations

In ALL_REVISIONS mode, the per-row transaction identifier is set on `LoadQueryInfluencers` during
`EntityInitializerImpl.resolveInstance()` and restored to `ALL_REVISIONS` during
`EntityInitializerImpl.finishUpRow()`. The restore was moved from `initializeInstance()` to
`finishUpRow()` (called by `StandardRowReader` after all phases complete) so that the temporal ID
remains set across all three row processing phases (`resolveKey` → `resolveInstance` → `initializeInstance`).

Row processing in `StandardRowReader.coordinateInitializers()` is **phase-based across all initializers**:
all `resolveKey()` calls first, then all `resolveInstance()`, then all `initializeInstance()`.
Within `resolveInstance()`, entity initializers are sorted first (via `InitializersList.sortedForResolveInstance`),
but their relative order among themselves is the insertion order from `AssemblerCreationStateImpl`.

Due to constructor recursion in `EntityInitializerImpl`, **child entity initializers (fetched associations)
are registered before their parent** — the `EntityInitializerImpl` constructor creates child fetch assemblers
(which register their own initializers) before the constructor returns and the parent is registered.

This means in `resolveInstance()`:

1. Child entity initializer runs first → reads its own REV column, sets temporal ID
2. Root entity initializer runs last → reads its own REV column, **overwrites** temporal ID

The root's REV is what survives into `initializeInstance()` and `finishUpRow()`.
This is correct because the root entity drives the temporal context: the child's REV is
derived from the root's via the correlated join predicate (`MAX(REV) WHERE REV \<= parent.REV`).

Currently, **both root and fetched entity initializers** select and read the `auditTransactionIdResult`
(the REV column from the audit table). This is set up in `AbstractEntityResultGraphNode.auditTransactionIdResult()`,
which is called for both `EntityResultImpl` (root results) and `EntityFetchJoinedImpl` (which creates its own
internal `EntityResultImpl`).

**Execution order (updated analysis):**

The root entity initializer's `resolveKey()` (called in phase 1 on `resultInitializers`) internally
calls `resolveInstance()` on itself (line 629) AND then `resolveKeySubInitializers()` which triggers
children's `resolveKey()`. So the actual execution order within a row is:

1. Root's `resolveEntityKey()` → creates root's `TemporalEntityKey` (reads REV from JDBC row directly)
2. Root's `resolveInstance()` → sets temporal ID to root's REV on `LoadQueryInfluencers`
3. Children's `resolveKey()` (via `resolveKeySubInitializers`) → creates children's `TemporalEntityKey`
4. Children's `resolveInstance()` (phase 2) → overwrites temporal ID with child's correlated REV

The temporal ID surviving into phase 3 (`initializeInstance`) is a child's REV (correlated to root's
via the join predicate). `TemporalEntityKey` creation is unaffected — both root and children read
directly from the JDBC row, independent of `LoadQueryInfluencers`.

**Known limitations and future considerations:**

* **Non-audited root with audited associations**: If the root (result) entity is not `@Audited` but one
  or more of its join-fetched associations are, the root has no `auditTransactionIdAssembler` and doesn't
  set the temporal ID. The child association initializer sets it during phase 2. This case needs
  investigation: does the child's temporal ID correctly drive association loading for sibling initializers?
  What is the expected behavior — should `ALL_REVISIONS` mode even be meaningful for a non-audited root?
* **Unnecessary child tx-id selection**: Currently both root and fetched entity initializers select and
  read the `auditTransactionIdResult` (via `AbstractEntityResultGraphNode`). Since the root sets the
  temporal ID first (during phase 1) and `isAllRevisions()` returns false after that (temporal ID is
  no longer `ALL_REVISIONS`), child initializers' ALL_REVISIONS path in `resolveInstance()` is not
  entered. This means the child tx-id selection and assembler creation may be unnecessary overhead —
  we should consider only selecting the tx-id for the root entity (identifiable via `isResultInitializer`).
* A more robust long-term approach: attach the temporal identifier to `TemporalEntityKey` (which already
  stores it) or `EntityHolder`, so initializers can read it from the entity they're associated with
  rather than from the shared mutable `LoadQueryInfluencers` field. This would eliminate ordering concerns.
* Collection initializers (not yet implemented for `@Audited`) will need similar analysis — their
  `initializeInstance()` runs after the entity initializer's, and the temporal ID must be correct at that point.
  The `finishUpRow()` approach ensures this.

A critical requirement: existing envers users must be able to migrate to core `@Audited` **without changing
their existing database schema**. Audit data is typically append-only and can span years — schema migration
is impractical or impossible for many users.

=== Key Insight: Schema Is Already Compatible

The good news: **core's audit table schema is a subset of envers' schema**. The defaults already match
(`REV`, `REVTYPE`, `_aud` suffix). The only real gap is the `REVINFO` table — envers creates it
automatically and FKs point to it, while core has no concept of it.

This gap is bridged by the `TransactionIdentifierSupplier` proposal: if the user maps their existing
`REVINFO` table as a regular `@Entity` and implements a supplier that persists to it, everything works:

1. The supplier persists a `RevisionInfo` entity → row written to `REVINFO` table
2. The generated ID is returned as the transaction identifier
3. Core writes this ID into the `REV` column of audit tables
4. The existing FK constraint (`*_AUD.REV` → `REVINFO.REV`) is satisfied
5. Core's `AuditMappingImpl.applyPredicate()` reads audit rows by `REV` value — works regardless of FK

=== Schema Compatibility Constraints

For the migration to work seamlessly, these constraints must hold:

1. **`REV` column type must match**: If envers used `INTEGER`, the supplier must return `Integer`.
   The `TransactionIdentifierService` must be configured with matching type.

2. **Column names must match**: Core defaults (`REV`, `REVTYPE`) already match envers defaults.
   If envers was configured with custom names (via `EnversSettings`), use `@Audited(transactionId=..., modificationType=...)`.

3. **Audit table names must match**: If envers used custom prefix/suffix, use `@Audited(tableName=...)`.

4. **REVINFO table must continue to be populated**: The `TransactionIdentifierSupplier` must persist
   to the same REVINFO table so FK constraints are satisfied and revision numbers continue the sequence.

5. **Sequence/generator must continue**: The `@GeneratedValue` on the REVINFO entity must use the same
   sequence/generator that envers used (e.g. `REVISION_GENERATOR` sequence).

=== What About ValidityAuditStrategy Users?

Core supports validity audit strategy via `hibernate.audit.strategy=validity`. Migration from envers:

- **Existing data**: Fully compatible. Core reads REVEND columns with the same range predicate as envers.
- **New writes**: Core populates REVEND on previous rows and writes new audit rows, same as envers.
- **Configuration**: Set `hibernate.audit.strategy=validity`. Column names default to `REVEND` and
  (optionally) `REVEND_TSTMP`, configurable via `@Audited.transactionEnd()` / `@Audited.transactionEndTimestamp()`.
- **Default strategy users**: No changes needed. The MAX-subquery approach (default) works on all data,
  including tables with REVEND columns (the extra columns are simply ignored).

=== Built-in REVINFO Support

`RevisionEntitySupplier` is a built-in `TransactionIdentifierSupplier` that handles the REVINFO
pattern out of the box. It persists a user-defined revision entity using a temporary child
`StatelessSession` (via `statelessWithOptions().connection().open()`) that shares the parent
session's JDBC connection and transaction, then returns the generated ID.

Key design decisions:

* **Child `StatelessSession` for persistence**: Avoids polluting the user's persistence context.
  The child session shares the same JDBC connection/transaction, so the revision entity insert
  participates in the user's transaction without interfering with the user's `Session`.
  This mirrors envers' approach of using `sessionWithOptions().connection()` for revision persistence.
* **On-demand initialization**: Transaction ID generation is lazy — the supplier is only called
  when `getCurrentTransactionIdentifier()` is first accessed (i.e. when an audited entity is
  actually modified). This prevents empty revisions for transactions that don't touch audited entities.
* **`RevisionListener` callback**: An optional callback for populating custom fields (user, timestamp, etc.)
  on the revision entity, invoked before persistence.

[source,java]
----
// Subclass approach:
public class RevisionInfoSupplier extends RevisionEntitySupplier<Integer> {
    public RevisionInfoSupplier() {
        super( RevisionInfo.class, rev -> {
            var revInfo = (RevisionInfo) rev;
            revInfo.timestamp = Instant.now().toEpochMilli();
            revInfo.username = SecurityContext.getCurrentUser();
        } );
    }
}
----

== Design Decisions

=== Decided

1. **Read paradigm**: Use standard session APIs with `atTransaction` context, not a custom query DSL.
   HQL already provides filtering, ordering, projections, joins, and aggregations.

2. **No built-in revision entity in core**: Transaction IDs are opaque and application-supplied.
   Users who need revision metadata (timestamp, user, etc.) define their own entity and persist it
   via `TransactionIdentifierSupplier`.

3. **Custom revision entity via `TransactionIdentifierSupplier`**: A session-aware supplier that can persist
   a revision entity and return its generated ID. This replaces envers' `@RevisionEntity` + `RevisionListener`
   with a single, simpler extension point.

=== To Decide

1. **Where does `AuditLog` live?**
   - Option A: In hibernate-core alongside `AuditMapping` (recommended — audit infra is already there)
   - Option B: In a slimmed-down hibernate-envers

2. **Transaction ID vs Revision Number semantics**
   - Envers: auto-incrementing integers with "next/previous" navigation
   - Core: arbitrary comparable types
   - Do we need navigation helpers? Recommendation: No — keep TX IDs opaque

3. **`AuditLog` query implementation**
   - Current: all queries use HQL via a lazily-created `ALL_REVISIONS` child session
     (shared JDBC connection, isolated PC, auto-closed when parent closes)
   - TODO: cache query plans to avoid HQL interpretation overhead on every call
     (see `AuditLogImpl` TODO comment)

4. **Criteria-based restrictions for `AuditLog` API**
   - Envers' `AuditQuery` accepted `AuditCriterion` restrictions (custom DSL: `AuditEntity.property("x").eq(...)`)
   - Core should use the standard JPA Criteria API instead of a custom DSL
   - Options for API shape:
     * Accept `CriteriaQuery` fragments / `Predicate` instances in `AuditLog` methods
     * Provide an `AuditLog.createQuery(Class)` returning a Criteria-like builder that
       pre-configures the all-revisions context and exposes `transactionId` / `modificationType`
     * Return a `CriteriaQuery` pre-configured for all-revisions mode that users can add predicates to
   - Key question: how to expose `transactionId()` and `modificationType()` in the Criteria API?
     These are HQL functions — they could be surfaced via `CriteriaBuilder.function()` or via
     dedicated helper methods on an `AuditLog`-specific criteria builder

4. **Modified property tracking (`withModifiedFlag`)**
   - Requires `_MOD` boolean columns per audited property
   - Significant schema + complexity cost
   - Defer in favor of `diff()` which computes changes from snapshots?

5. **Fate of hibernate-envers module**
   - Option A: Deprecate entirely, provide migration guide
   - Option B: Rewrite as thin adapter over core `@Audited` + `AuditLog`
   - Recommendation: Option B for transition, then A long-term

6. **Deleted entity access** — [green]#DECIDED#: `atTransaction` returns null for deleted entities
   (correct point-in-time semantics). `ALL_REVISIONS` mode provides raw access including DEL rows.
   `AuditLog.find()` filters out deletions via `modificationType(e) != DEL`.
   `AuditLog.getHistory()` includes all modification types.

=== Remaining Gaps

[cols="1,3,1,3"]
|===
| # | Gap | Priority | What's Needed

| G5
| **Cross-type revision queries**
| Medium
| Methods on `AuditLog`: `getEntityTypesModifiedAt(txId)`, `findAllEntitiesModifiedAt(txId)`.
  Envers requires opt-in (`track_entities_changed_in_revision=true`) which populates a
  `REVCHANGES` index table via `EntityTrackingRevisionListener`; without it,
  `getCrossTypeRevisionChangesReader()` throws. The envers implementation queries REVCHANGES
  first (1 query), then fetches per-type snapshots (n queries).
  Core can implement naively by scanning all audited persisters (N queries, no index table),
  but for efficiency at scale an opt-in write-side index (via G9) should be supported later.

| G7
| **`RelationTargetAuditMode.NOT_AUDITED`**
| [green]#DONE#
| Works implicitly — no annotation needed. FK columns are copied to audit table;
  non-audited targets load from current table via existing null guards on
  `getAuxiliaryMapping()`. Envers required explicit `@Audited(targetAuditMode = NOT_AUDITED)`
  because its separate metadata model needed declarations; core reuses entity mappings.
  Tested: `@ManyToOne`, `@OneToOne`, `@ManyToMany`, `@OneToMany` to non-audited targets.
  See <<not-audited-target-behavior>> for behavioral differences with deleted targets.

| G8
| **`@Version` auto-exclusion**
| Medium
| Envers excludes `@Version` by default (`do_not_audit_optimistic_locking_field=true`).
  Core includes it (users can use `@Audited.Excluded`).
  Auto-exclusion requires coordinating boot-time column exclusion, write-side mask, AND
  read-side SQL generation — non-trivial.

| G9
| **`EntityTrackingRevisionListener`** (per-entity-change callback)
| [green]#DONE#
| `o.h.audit.EntityTrackingRevisionListener extends RevisionListener` with
  `entityChanged(Class, String, Object, ModificationType, Object revisionEntity)`.
  Called once per merged entity change from `AuditWorkQueue.doBeforeTransactionCompletion()`,
  inline with the entity write loop (single iteration, null-check guarded by JIT).
  Revision entity stored on the work queue by `RevisionEntitySupplier` — no re-fetch needed.

| G10
| **`@AuditOverride`** for inherited properties
| Low
| Override audit settings on inherited properties from `@MappedSuperclass` or superclasses.
  `@Audited.Excluded` covers the common case; the `forClass` pattern has no equivalent.

| G11
| **Criteria-based restrictions for `AuditLog`**
| Low
| Consider accepting JPA `Predicate` fragments. HQL is a viable workaround.

| G12
| **Audit table suffix case**: envers `_AUD` vs core `_aud`
| Medium
| On case-insensitive databases (most), transparent. On case-sensitive databases
  (e.g. PostgreSQL with quoted identifiers), migration from envers requires matching.
  Options: make suffix configurable, or match envers default `_AUD`.

| G14
| **Audit table schema/catalog override**
| Medium
| No way to put audit tables in a different schema/catalog.
  Envers supports `default_schema`/`default_catalog` settings and per-annotation
  `schema`/`catalog` attributes. Common use case: separate audit schema for access control.
  Needs attributes on `@Audited` and/or a global configuration property.

| G16
| **Foreign key constraints on audit tables**
| High
| Core audit tables currently have NO FK constraints. Envers created:
  (1) `_AUD.REV -> REVINFO.REV` on every audit table;
  (2) `_AUD.REVEND -> REVINFO.REV` for validity strategy;
  (3) `ChildEntity_AUD.(REV, id) -> ParentEntity_AUD.(REV, id)` for JOINED inheritance;
  (4) `secondary_AUD.(REV, id) -> PrimaryEntity_AUD.(REV, id)` for `@SecondaryTable`.
  No FK from `_AUD` back to the main entity table. No unique constraints (composite PKs
  handle uniqueness). These FKs are important for data integrity and schema compatibility
  with existing envers databases.

| G18
| **Pre-built SQL AST for `getHistory()`**
| Medium
| `AuditLog.find()` uses pre-built SQL AST via `AuditFindPlan` (zero HQL parsing).
  `getHistory()` still uses HQL because building a multi-entity `DomainResult`
  (audit entity + revision entity join) within a single `LoaderSqlAstCreationState`
  causes SQL selection position mismatches when the revision entity's `DomainResult`
  reads from wrong JDBC result positions. A `PostProcessor` callback was added to
  `LoaderSelectBuilder.generateSelect()` to allow adding extra joins/DomainResults
  using the same creation state, but the revision entity ID column still resolves
  to the wrong position (reads audit entity's BIGINT id instead of REVINFO's INT rev).
  The SQL output is correct — the issue is in the `DomainResult` internal position
  mapping. Further investigation needed. The `PostProcessor` API on `LoaderSelectBuilder`
  is a useful extension that should be kept regardless.

| G17
| **`AuditLog` API: builder pattern for entity-scoped queries**
| Medium
| Currently every method on `AuditLog` is duplicated for `Class<T>` and `String entityName`
  parameters. Should refactor to a fluent builder pattern:
  `auditLog.forEntity(MyEntity.class).find(id, txId)` / `auditLog.forEntity("MyEntity").find(...)`,
  similar to `session.byId(Class)` returning `IdentifierLoadAccess`. Eliminates the
  duplication and provides a clear path for future methods.

| G15
| **Naming: "revision" vs "transaction-id"**
| Low
| Core uses "transaction-id" naming in the new API (`transactionId()`, `getTransactionId()`,
  `@Audited.transactionId`), but several internal classes retain the "revision" naming from envers:
  `RevisionEntitySupplier`, `RevisionListener`, `@RevisionEntity`, `@RevisionNumber`,
  `@RevisionTimestamp`, `revisionNumberProperty`, etc. Need to decide whether to fully migrate
  these to "transaction-id" naming or preserve "revision" for backward compatibility with
  the envers conceptual model. The "revision" naming is arguably clearer for audit-specific
  concepts (a "revision" is an audit-specific notion, while "transaction-id" is more general).

| G6
| **ValidityAuditStrategy** (REVEND / REVEND_TSTMP columns)
| [green]#DONE#
| `hibernate.audit.strategy=validity` + `@Audited(transactionEnd, transactionEndTimestamp)`.
  REVEND on root, TPC subclass, and collection audit tables (not secondary, not JOINED —
  matching envers). Range-based read predicate. Per-element REVEND UPDATE for collections.
  Deferred audit writes via `AuditWorkQueue` prevent multi-flush PK violations.
  `@AuditedTest` runs all tests with both strategies.
  Minor difference from envers: `allow_identifier_reuse` not exposed as a setting — ID reuse
  is handled implicitly (REVEND UPDATE for ADD accepts 0 or 1 affected rows).

| Significant schema + complexity. `diff()` alternative is simpler. Opt-in in envers too.

| `AuditStrategy` SPI (pluggable)
| Premature until ValidityAuditStrategy is implemented.

| `CASCADE_DELETE_REVISION`
| Anti-pattern for audit data.

| Revision number navigation (next/prev)
| Opaque IDs; query `getRevisions()` and navigate the list.

| `NotAuditedException` / `RevisionDoesNotExistException`
| Can add specific exception types later if needed.

| `@ModifiedEntityNames` / `REVCHANGES` table
| Depends on G9 + G5. Opt-in feature, implementable later.

== ValidityAuditStrategy Design

=== Overview

ValidityAuditStrategy adds a `REVEND` column to audit tables that marks when an audit row was
superseded. This enables range-based point-in-time queries instead of MAX(REV) subqueries:

[source,sql]
----
-- Default strategy (current): MAX subquery
WHERE REV = (SELECT MAX(REV) FROM aud WHERE REV <= :txId AND id = :id)

-- Validity strategy: range predicate (much faster, no subquery)
WHERE REV <= :txId AND (REVEND > :txId OR REVEND IS NULL)
----

Write-side cost: one additional UPDATE per audited mutation (set `REVEND` on the previous row).

=== Configuration

Global property `hibernate.audit.strategy` with values `default` (MAX subquery) and `validity`
(REVEND range). Default: `default`. Added to `StateManagementSettings`.

`@Audited` annotation gets new attributes:

* `transactionEnd()` — REVEND column name (default `"REVEND"`)
* `transactionEndTimestamp()` — optional REVEND_TSTMP column name (default `""`, disabled)

=== Schema

When validity strategy is active, each audit table gets additional columns:

* `REVEND` — same type as REV, **nullable** (NULL = current/latest row)
* `REVEND_TSTMP` (optional) — timestamp of when the row was superseded

=== Read-Side

Single branch point in `AuditMappingImpl.createRestriction()`: if
`TableAuditInfo.revisionEndMapping` is non-null, build range predicate instead of MAX subquery.
The validity predicate is simpler (no subquery, no key correlation):

[source,java]
----
// REV <= :txId
// AND (REVEND > :txId OR REVEND IS NULL)
// AND REVTYPE <> DEL (when applicable)
----

All five `applyPredicate()` overloads are untouched — they delegate to `createRestriction()`.
ALL_REVISIONS mode is also untouched — `hasTemporalPredicate()` returns false, so no predicate
is applied regardless of strategy.

=== Write-Side

`AbstractAuditCoordinator` builds a cached UPDATE mutation group:
`SET REVEND = :txId [, REVEND_TSTMP = :tstmp] WHERE id = :id AND REVEND IS NULL AND REV <> :txId`.
Called after `insertAuditRow()` in all entity coordinators (`Insert`, `Update`, `Delete`).
Direct execution (no transaction callback) since core coordinators are inline.

For collection audit tables, same pattern: UPDATE previous row's REVEND after inserting new
audit rows. WHERE clause uses all key columns (FK + element/index columns).

For JOINED inheritance, one UPDATE per audit table in the hierarchy (each has its own REVEND).

=== `TableAuditInfo` Extension

[source,java]
----
public record TableAuditInfo(
    String auditTableName,
    SelectableMapping transactionIdMapping,
    @Nullable SelectableMapping modificationTypeMapping,
    @Nullable SelectableMapping revisionEndMapping,
    @Nullable SelectableMapping revisionEndTimestampMapping
) {}
----

For default strategy, `revisionEndMapping` is null — existing code paths are unaffected.

=== Design Consideration: AuditStrategy as SPI

Envers' `AuditStrategy` was an SPI (`org.hibernate.envers.strategy.spi.AuditStrategy`), allowing
integrators to contribute custom strategies that modify the audit schema (`addAdditionalColumns()`)
and define behavior for persisting (`perform()`) and reading (`addEntityAtRevisionRestriction()`)
audit data. This enabled third-party strategies beyond Default and Validity.

Core's current design hardcodes the strategy choice into `AuditMappingImpl.createRestriction()`
(branching on `revisionEndMapping != null`) and `AbstractAuditCoordinator` (REVEND UPDATE logic).
A custom strategy cannot be plugged in without modifying these classes.

**Options for the future:**

1. **Extract a strategy interface** in `org.hibernate.audit.spi` with methods for:
   - `addAdditionalColumns(AuditTableContext)` — boot-time schema contribution
   - `createRestriction(TableReference, TableAuditInfo, Expression)` — read-side predicate
   - `updatePreviousRow(...)` — write-side lifecycle hook
   This would mirror envers' SPI but adapted to core's SQL AST model.

2. **Keep it internal** — the two built-in strategies (default + validity) cover the vast majority
   of use cases. A formal SPI adds maintenance burden and compatibility constraints. Can be
   introduced later if third-party strategies emerge as a real need.

**Current decision**: keep internal (option 2). The `hibernate.audit.strategy` property and
`@Audited.transactionEnd()` annotation are the user-facing configuration points. The implementation
is internal and can be refactored into an SPI later without breaking public API.

== Open Questions

=== Resolved

* [green]#RESOLVED#: **Relationship auditing** — ManyToOne fully tested (eager, lazy, nested).
  OneToOne (FK-based) verified. OneToMany/ManyToMany still TODO (see G2/G3).
* [green]#RESOLVED#: **Embeddable auditing** — simple `@Embedded` works. Nested embeddables likely work.
  `@EmbeddedId` works (was blocked by reserved keyword in test).
* [green]#RESOLVED#: **`@Version` fields** — included in audit tables by default. Users can use
  `@Audited.Excluded`. Auto-exclusion is non-trivial (requires coordinating 3 layers).
* [green]#RESOLVED#: **Audited associations** — work transparently via `TemporalEntityKey`,
  per-row temporal identifier, and lazy proxy temporal capture.

=== Still Open

* **`ALL_REVISIONS` for `@Temporal` entities**: Currently `ALL_REVISIONS` is audit-specific and
  `TemporalMappingImpl` explicitly ignores it (guards in `useAuxiliaryTable`, `isAffectedByInfluencers`,
  `useTemporalRestriction`). Extending it to `@Temporal` would require handling different strategies
  (SINGLE_TABLE: remove predicate; HISTORY_TABLE: union current + history; NATIVE: `FOR SYSTEM_TIME ALL`)
  and using `row_start` as the `TemporalEntityKey` discriminator (assumed unique at sufficient precision).
  Deferred — see discussion on `row_start` uniqueness guarantees.
* [green]#RESOLVED#: **JOINED inheritance read side**: Multi-table `AuditMappingImpl` with per-table
  `TableAuditInfo` map. Per-table temporal predicates on joined table references. ALL_REVISIONS
  REV correlation. Subclass persisters inherit root's multi-table mapping via `AuditStateManagement`.
* [green]#RESOLVED#: **TABLE_PER_CLASS deep hierarchy**: Pre-built audit union subqueries stored in
  `TableAuditInfo` map; MAX(REV) subquery uses `getTableName()` (returns union subquery for
  persisters with subclasses) to span all audit tables. `toAuditSubquery()` eliminated.
* **Associations to polymorphic entities**: Verify that to-one and to-many associations pointing
  to entities with SINGLE_TABLE, JOINED, and TABLE_PER_CLASS inheritance work correctly in
  point-in-time reads.
* **`REVISION_ON_COLLECTION_CHANGE`**: Investigated. Envers wrote MOD rows for inverse (mappedBy)
  entities when the owning side's collection changed, via `generateBidirectionalCollectionChangeWorkUnits()`
  in `BaseEnversEventListener`. This applied to both bidirectional `@OneToMany` and `@ManyToMany`.
  Envers used `AuditProcess` work-unit buffering with merge semantics for deduplication
  (ADD+CollectionChange→ADD, MOD+CollectionChange→single MOD, etc.). **Decision**: not replicated
  in core — the relationship history is fully captured on the owning side. Can be revisited if users
  need `getRevisions()` on inverse entities to include relationship changes.
* [green]#RESOLVED#: **ValidityAuditStrategy**: Implementing in core (G6). See <<ValidityAuditStrategy Design>>.
* **`EntityTrackingRevisionListener`** (G9): Per-entity-change callback. Prerequisite for
  `@ModifiedEntityNames` and efficient cross-type queries (G5). The callback should fire from
  mutation coordinators. Design question: should it receive the revision entity (like envers)
  or just the transaction ID?
* **Cross-type revision queries** (G5): Naive scan iterates all audited persisters (N queries).
  For efficient cross-type queries at scale, opt-in write-side tracking (like envers' `REVCHANGES`)
  is needed, tied to G9. Both approaches should be available.
* **Audit table schema/catalog** (G14): Separate schema for audit tables. Needs `@Audited`
  attributes and/or a global configuration property.
* **Audit table suffix case** (G12): envers `_AUD` vs core `_aud`. Make configurable or
  match envers default for migration compatibility.
* **Performance**: `getRevisions()` pagination for entities with many revisions.
* **Envers `originalId` mapping**: Verify this doesn't affect core's column-level reads.
* **Sequence continuity**: Document that `@GeneratedValue` must use the same sequence as envers
  (`REVISION_GENERATOR`) for migration.
