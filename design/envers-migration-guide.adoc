= Migration Guide: Envers to Core @Audited
:toc:
:toclevels: 2

== Overview

Hibernate Core integrates audit trail persistence directly via `@org.hibernate.annotations.Audited`,
replacing the separate `hibernate-envers` module. The core approach is fundamentally simpler:
**"open a session at a point in time, use normal Hibernate."** Instead of a dedicated audit query
DSL (`AuditQuery`, `AuditEntity`), you use standard HQL, Criteria, and `session.find()` within
a temporal session. The `AuditLog` API provides convenience methods for common audit metadata queries.

This guide covers the migration path from envers to core `@Audited`.

See also: link:envers-read-rewrite.adoc[Envers Rewrite -- Design & Implementation] for the
full design document.

== Legacy Bridge

A legacy `AuditReader` interface is provided in `org.hibernate.audit.legacy` that extends
`AuditLog` with deprecated bridge methods matching the envers `AuditReader` API. Most
envers code works with just an import change from `org.hibernate.envers` to
`org.hibernate.audit.legacy`.

`AuditReaderFactory.get(session)` returns the session's `AuditLog` (which implements
`AuditReader`) directly. No adapter class is needed.

Unsupported methods (`createQuery()`, `getCrossTypeRevisionChangesReader()`,
`getCurrentRevision()`, `getEntityName()`) throw `UnsupportedOperationException`
with migration hints.

== Breaking Changes

* `getRevisions()` returns `List<Object>` instead of `List<Number>`.
  Update variable declarations from `List<Number>` to `List<Object>`.
* `findRevisions()` returns `Map<Object, T>` instead of `Map<Number, T>`.
  Update variable declarations accordingly.
* These changes reflect that core transaction identifiers are `Object`-typed
  (not restricted to `Number`).

| `REV`, `REVTYPE` (configurable via `@Audited`)
| Yes -- defaults match

| Composite PK: `(REV, entity_id)` for entities; `(REV, all_source_cols)` for collections

=== Key Insight: Schema Is Already Compatible

Core's audit table schema is a subset of envers' schema. The defaults already match
(`REV`, `REVTYPE`, `_aud` suffix). The only real gap is the `REVINFO` table -- envers creates it
automatically and FKs point to it, while core only creates it when `@RevisionEntity` is configured.

This gap is bridged by mapping the existing REVINFO table as a `@RevisionEntity`:

1. The supplier persists a `RevisionInfo` entity -> row written to `REVINFO` table
2. The generated ID is returned as the transaction identifier
3. Core writes this ID into the `REV` column of audit tables
4. The existing FK constraint (`*_AUD.REV` -> `REVINFO.REV`) is satisfied
5. Core's `AuditMappingImpl.applyPredicate()` reads audit rows by `REV` value -- works regardless of FK

=== Migration Recipe

==== Step 1: Map the existing REVINFO table

[source,java]
----
// Map the EXISTING REVINFO table -- no schema changes needed
@Entity
@Table(name = "REVINFO")
@RevisionEntity(listener = MyRevisionListener.class)
public class RevisionInfo {
    @Id
    @GeneratedValue  // matches envers' generator strategy
    @RevisionNumber
    @Column(name = "REV")
    private int id;

    @RevisionTimestamp
    @Column(name = "REVTSTMP")
    private long timestamp;

    // Add any custom fields that were in the user's @RevisionEntity
    // private String username;

    // getters/setters
}
----

==== Step 2: Implement `RevisionListener` (optional)

[source,java]
----
public class MyRevisionListener implements RevisionListener {
    @Override
    public void newRevision(Object revisionEntity) {
        var rev = (RevisionInfo) revisionEntity;
        // rev.setUsername(SecurityContext.getCurrentUser());
    }
}
----

==== Step 3: Replace envers annotations

[source,java]
----
// Before (envers):
@org.hibernate.envers.Audited
@Entity
public class Customer { ... }

// After (core):
@org.hibernate.annotations.Audited
@Entity
public class Customer { ... }
----

==== Step 4: Remove envers integration

[source,properties]
----
# Remove envers integration
# hibernate.integration.envers.enabled=true  <- remove

# Audit strategy (if using ValidityAuditStrategy)
# hibernate.audit.strategy=validity
----

==== Step 5: Read historical data

[source,java]
----
// Before (envers):
AuditReader reader = AuditReaderFactory.get(entityManager);
Customer c = reader.find(Customer.class, customerId, revisionNumber);

// After (core) -- same revision numbers work because same REVINFO table:
try (var s = sf.withOptions().atTransaction(revisionNumber).open()) {
    Customer c = s.find(Customer.class, customerId);
}

// Or using AuditLog:
var auditLog = session.getAuditLog();
Customer c = auditLog.find(Customer.class, customerId, revisionNumber);
----

=== Schema Compatibility Constraints

For the migration to work seamlessly, these constraints must hold:

1. **`REV` column type must match**: If envers used `INTEGER`, the supplier must return `Integer`.
   The `TransactionIdentifierService` must be configured with matching type.

2. **Column names must match**: Core defaults (`REV`, `REVTYPE`) already match envers defaults.
   If envers was configured with custom names (via `EnversSettings`), use `@Audited(transactionId=..., modificationType=...)`.

3. **Audit table names must match**: If envers used custom prefix/suffix, use `@Audited(tableName=...)`.

4. **REVINFO table must continue to be populated**: The `@RevisionEntity` must map
   to the same REVINFO table so FK constraints are satisfied and revision numbers continue the sequence.

5. **Sequence/generator must continue**: The `@GeneratedValue` on the REVINFO entity must use the same
   sequence/generator that envers used (e.g. `REVISION_GENERATOR` sequence).

=== ValidityAuditStrategy Users

Core supports validity audit strategy via `hibernate.audit.strategy=validity`. Migration from envers:

- **Existing data**: Fully compatible. Core reads REVEND columns with the same range predicate as envers.
- **New writes**: Core populates REVEND on previous rows and writes new audit rows, same as envers.
- **Configuration**: Set `hibernate.audit.strategy=validity`. Column names default to `REVEND` and
  (optionally) `REVEND_TSTMP`, configurable via `@Audited.transactionEnd()` / `@Audited.transactionEndTimestamp()`.
- **Default strategy users**: No changes needed. The MAX-subquery approach (default) works on all data,
  including tables with REVEND columns (the extra columns are simply ignored).

=== Built-in REVINFO Support

`RevisionEntitySupplier` is a built-in `TransactionIdentifierSupplier` that handles the REVINFO
pattern out of the box. It persists a user-defined revision entity using a temporary child
`StatelessSession` (via `statelessWithOptions().connection().open()`) that shares the parent
session's JDBC connection and transaction, then returns the generated ID.

Key design decisions:

* **Child `StatelessSession` for persistence**: Avoids polluting the user's persistence context.
  The child session shares the same JDBC connection/transaction, so the revision entity insert
  participates in the user's transaction without interfering with the user's `Session`.
  This mirrors envers' approach of using `sessionWithOptions().connection()` for revision persistence.
* **On-demand initialization**: Transaction ID generation is lazy -- the supplier is only called
  when `getCurrentTransactionIdentifier()` is first accessed (i.e. when an audited entity is
  actually modified). This prevents empty revisions for transactions that don't touch audited entities.
* **`RevisionListener` callback**: An optional callback for populating custom fields (user, timestamp, etc.)
  on the revision entity, invoked before persistence.

[source,java]
----
// Using @RevisionEntity annotation (auto-detected):
@Entity
@RevisionEntity(listener = MyRevisionListener.class)
public class MyRevision {
    @Id @GeneratedValue
    @RevisionNumber
    private int id;

    @RevisionTimestamp
    private Instant timestamp;

    private String username;
}

public class MyRevisionListener implements RevisionListener {
    @Override
    public void newRevision(Object revisionEntity) {
        var rev = (MyRevision) revisionEntity;
        rev.setUsername(SecurityContext.getCurrentUser());
    }
}
----

== Behavioral Differences

[[not-audited-target-behavior]]
=== Associations to Non-Audited Targets

Core auditing handles mixed audited/non-audited entity relationships implicitly -- no annotation
needed (unlike envers' `@Audited(targetAuditMode = NOT_AUDITED)`). The FK is stored in the audit
table and used to load the target from the current table. However, there are behavioral differences
with envers when the non-audited target entity has been **deleted**:

[cols="1,2,2"]
|===
| Scenario | Envers behavior | Core behavior

| **Eager / `optional=true`**
| `targetNotFoundAction` controls: ERROR (default) throws, IGNORE returns null
| LEFT JOIN resolves to null silently (standard Hibernate semantics)

| **Lazy + deleted target**
| Proxy created; on access, `targetNotFoundAction` controls: ERROR throws `EntityNotFoundException`,
  IGNORE eagerly loads and returns null (no proxy)
| Proxy created; on access, throws `EntityNotFoundException` (standard Hibernate proxy behavior)

| **`optional=false` + deleted target**
| ERROR throws, IGNORE returns null
| INNER JOIN fails to load the owning entity itself (returns null from `find()`)
|===

**Root cause**: envers had its own entity reconstruction pipeline (field-by-field mapping from audit
row data, separate per-association loading via `ToOneEntityLoader`). Core reuses standard Hibernate
entity loading with JOIN fetching, so association behavior follows standard Hibernate semantics.

**Note**: audit tables for audited-to-non-audited associations have no FK constraints to the
non-audited target table (same as envers), so dangling FKs are always possible.
For audited-to-audited associations, envers' `AuditReaderImpl.find()` silently returned null for
missing targets (`NoResultException` caught). Core's JOIN-based approach handles this via the
temporal predicate + LEFT JOIN.

**Migration options**:

1. **Recommended**: migrating users should annotate audited-to-non-audited lazy associations with
   `@NotFound(action = IGNORE)` or `@NotFound(action = EXCEPTION)` to explicitly declare the
   expected behavior when the target is deleted.
2. **Future**: core could implicitly default to `@NotFound(EXCEPTION)` or `@NotFound(IGNORE)` for
   associations from audit tables to non-audited entities, matching envers' `targetNotFoundAction`
   default. This would require changes to how audit table references handle association fetching.

|===

| `@AuditOverride`

| `@ModifiedEntityNames`
| Use `EntityTrackingRevisionListener` to track manually

== Envers Annotations -- Full Status

[cols="2,1,3"]
|===
| Envers Annotation | Status | Core Status / Migration Path

| `@Audited(targetAuditMode, targetNotFoundAction, withModifiedFlag, modifiedColumnName)`
| [green]#DONE#
| `@o.h.annotations.Audited`. Has `tableName`, `transactionId`, `modificationType`, `Excluded`.
  Missing attributes: `withModifiedFlag` (deferred),
  `targetNotFoundAction` (low priority).

| `@NotAudited`
| [green]#DONE#
| `@Audited.Excluded` -- different mechanism (nested annotation), equivalent functionality.

| `@AuditTable(value, schema, catalog)`
| [yellow]#PARTIAL#
| `@Audited(tableName=...)` covers the table name. Missing: `schema`, `catalog`.

| `@RevisionEntity` / `@RevisionNumber` / `@RevisionTimestamp`
| [green]#DONE#
| Equivalent annotations in `o.h.audit`.

| `@AuditOverride` / `@AuditOverrides`
| [red]#TODO#
| `@Audited.Excluded` covers the common case (un-audit inherited property).
  The `forClass` pattern (override without re-declaring) has no equivalent.

| `@AuditJoinTable` / `@CollectionAuditTable`
| [yellow]#PARTIAL#
| `@Audited(tableName=...)` covers table name. Schema/catalog not yet supported.
  `inverseJoinColumns` customization not supported. Low priority.

| `@SecondaryAuditTable` / `@SecondaryAuditTables`
| [yellow]#PARTIAL#
| Secondary audit tables are auto-created. Custom naming not yet supported.

| `@AuditMappedBy(mappedBy, positionMappedBy)`
| [green]#N/A#
| Core uses the same entity mappings (not separate audit entities), so the FK column
  and relationship are resolved by Hibernate's metamodel. No equivalent annotation needed.
  (Experimental in envers.)

| `@ModifiedEntityNames`
| [yellow]#DEFERRED#
| Depends on cross-type queries. Backed by `REVCHANGES` table. Opt-in in envers.
|===
